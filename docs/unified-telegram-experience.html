🚫 Before: Two Paradigms

Users had to think about modes and commands

  • → /claude for simple tasks
  • → /liaison for complex orchestration
  • → Different UIs for different modes
  • → Reply to continue vs. button callbacks
  • → Separate mental models

✨ After: One Experience

The system adapts; users just talk

  • ✓ Natural language, no commands needed
  • ✓ Session cards scale from 1 to N agents
  • ✓ Same UI whether simple or complex
  • ✓ Consistent reply & button interactions
  • ✓ One mental model: "I'm talking to Takopi"

Unified Design Principles

🎯

Progressive Disclosure

Simple tasks look simple. Expand to see complexity only when you want it.

💰

Consistent Controls

Pause, resume, watch, continue - same buttons everywhere, same behavior.

💬

Conversational First

Reply to any message naturally. The system figures out the routing.

👀

Transparent Orchestration

See which agents are active, but don't manage them. Trust the liaison.

⚡

Smart Escalation

Only interrupted for decisions that matter. Routine stuff is handled.

🔄

Seamless Continuity

Every session is resumable. Pick up where you left off, anytime.

Implementation Notes

Session detection: Takopi analyzes the request and decides whether to use direct engine (simple) or liaison (complex). The user never chooses.

Unified routing: All replies flow through the same handler. If there's an active session, it continues. If there's a pending question, it answers it. Otherwise, it starts fresh.

Agent badges: The session card shows active agents dynamically. A single-agent task shows one badge. Multi-agent shows multiple. Same component, same UI.
